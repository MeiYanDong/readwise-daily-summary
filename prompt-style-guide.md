# 提示词写作规范

本项目所有提示词以独立 Markdown 文件存放，方便在 Obsidian 中查看修改，也便于复刻到其他项目。

---

## 文件结构

每个提示词文件按以下章节顺序组织：

```
# [功能名] — 提示词

## 角色/role
## 任务/task
## 核心要求/requirements  （或 核心原则/principles）
## 输出格式/format
## 注意事项/notes          （可选）
```

章节标题中英双写，方便中英文环境都能理解。

---

## 各章节写法

### 角色/role

两件事：**身份定义** + **读者画像**。

- 身份要有专业感，但不要泛化。不写"你是一个 AI 助手"，写"你是 Virtuals Protocol 的生态观察者"
- 读者画像要具体到人，不写"用户"，写"在校大学生和年轻开发者，热爱 vibe coding，用 Claude Code 做项目、写作业、搞副业"
- 读者画像决定语气和信息密度，是整个提示词最重要的一句话

### 任务/task

一句话说清楚要做什么，不展开。展开放到后面的章节。

### 核心要求/requirements

用加粗 bullet 列约束，每条一个原则，不混合：

```markdown
- **按主题聚类**：将内容按话题归类，不按来源罗列
- **区分信号与噪声**：突出有信息量的内容，忽略纯广告、招聘、重复信息
```

**反面约束要明确写出来**，比正面约束更有效：

- ✓ 不要写"建议阅读官方文档"这种废话
- ✓ 不要罗列 changelog，只写真正影响使用体验的功能
- ✓ 不要硬凑，没有好内容直接省略板块

### 输出格式/format

用代码块包裹完整示例，占位符用 `[方括号]`：

````markdown
## 输出格式/format

```
# [日期] AI 日报

## 📌 今日值得细读
- **[标题]** — 一句话说明价值 → [链接](url)

## [主题一]
- 要点 → [链接](url)
```
````

让 LLM 直接对照示例生成，比用文字描述格式更可靠。

### 注意事项/notes

边界情况和特殊处理规则。重点写两类：

1. **内容类型的特殊处理**：如"Twitter List 聚合帖只提取有信息量的推文"、"SEC 文件直接跳过"
2. **内容不足时的降级行为**：要明确说清楚，不能让 LLM 自己决定

```markdown
- 如果某天 feed 内容很少，如实说明即可，不需要硬凑
- 如果某个板块本周没有好内容，直接省略
```

---

## 排序逻辑（适用于追踪类提示词）

当输出需要排优先级时，按**决策价值**从高到低排序，而不是按时间或来源：

1. **改变故事** — 新赛道、重大合作、协议级变化
2. **证明在前进** — 可量化的进展、里程碑达成
3. **值得留意** — 有潜力但需要后续验证的信号
4. **背景补充** — 丰富理解但不紧急的信息

## 语气规则

| 避免 | 改用 |
|---|---|
| 用户 | 你 |
| 建议使用 | 试试 |
| 圈内黑话（alpha、thesis、narrative） | 普通人能懂的表达 |
| 技术文档语气 | 懂技术的朋友在群里分享 |
